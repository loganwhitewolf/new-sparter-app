<objective>
Assemble the standalone discovery service `lib/services/regex-discovery.ts`. `discoverRegexCandidates({ userId, scope: { platformId } })` fetches Set B via the Wave-1 DAL query, applies the platform `descriptionStripPattern` strip followed by `normalizeDescription`, loads active patterns for coverage filtering, then delegates clustering to the Wave-1 `detectPatternSuggestionsWithMeta` and returns enriched candidates plus counts. This delivers PIPE-02 (standalone, import-independent service), completes PIPE-03 (normalization applied before discovery + per-candidate D-05 report), honors D-01 (operates on the persisted post-categorization Set B) and D-03 (platform-scoped read input via `scope.platformId`), and anchors Success Criterion 4 (the Fineco DoD input reaches discovery as normalized, uncategorized text). The legacy pre-categorization call in `analyzeFile` is annotated with a removal TODO and left intact so the current import summary keeps working until Phase 55 cuts over.

Purpose: This is the integration layer locked by D-02 — action → service → util, with the service owning normalization (strip + normalizeDescription) and orchestration, the DAL owning the fetch, and the util owning pure clustering. The service signature `{ userId, scope: { platformId } }` is shaped to support both Phase 54 entry points (post-import auto-run + Files-table on-demand) without building them here.

Output: New file `lib/services/regex-discovery.ts` (`discoverRegexCandidates`, `DiscoveryScope`, `DiscoveryResult`), a service unit test including the Fineco DoD anchor, and a one-line TODO comment in `lib/services/import.ts`.
</objective>
