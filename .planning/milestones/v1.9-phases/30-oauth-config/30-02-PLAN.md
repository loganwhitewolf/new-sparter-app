<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| process env → Better Auth config | Untrusted operator-provided env values (`GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET`) cross into the Better Auth runtime config at server startup |
| OAuth provider → `/api/auth/callback/{provider}` | Untrusted callback responses from Google / GitHub reach the Better Auth handler |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-30-07 | Spoofing (S) | OAuth callback to `/api/auth/callback/google` and `/.../github` | mitigate | Better Auth library handles state/PKCE for both providers internally — no custom callback logic introduced; the destructured `toNextJsHandler(auth)` route (Plan 01) preserves the library's CSRF/state checks. |
| T-30-08 | Tampering (T) | `socialProviders` config object | mitigate | The conditional spread uses pure `process.env` reads with no string interpolation, JSON parse, or eval. A missing `*_CLIENT_ID` produces an empty `{}` spread — cannot be coerced into a partial provider entry. |
| T-30-09 | Information disclosure (I) | `clientSecret` exposure | mitigate | `clientSecret` is read directly from `process.env.*` and passed to Better Auth; never logged, never serialized, never returned in any response. Operator runbook (Plan 03) flags these as secret-class variables. The `!` non-null assertion produces a runtime crash if the secret is missing (loud failure, no silent fallback). |
| T-30-10 | Elevation of privilege (E) | first-time OAuth sign-in creating a user with `role: 'user'` | mitigate | `auth.ts` `additionalFields.role` defaults to `'user'` with `input: false` (unchanged). OAuth-provisioned users inherit this default — cannot self-assign `admin`. `subscriptionPlan` defaults to `'free'` with `input: false`. |
| T-30-11 | Tampering (T) | accidental provider mis-pairing (e.g. only `GOOGLE_CLIENT_ID` set, `GOOGLE_CLIENT_SECRET` missing) | accept | The `!` on `clientSecret` produces an `undefined` value at runtime, which Better Auth will reject loudly at the first OAuth attempt. The `.env.example` comment (Plan 03) documents the pairing requirement. Acceptable — operator-only risk, no security degradation. |
</threat_model>
