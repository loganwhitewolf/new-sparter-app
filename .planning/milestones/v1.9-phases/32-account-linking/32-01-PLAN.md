      ))}

  )
}

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Next.js server (App Router) | Each /settings/* request is gated by `proxy.ts` session check; the redirect from /profile preserves this gate because the redirect target /settings/profile is also behind `proxy.ts`. |
| Server component env -> client component prop | `SettingsHub` is a server component; it does not pass any user-scoped data. Topbar already reads session via `authClient.useSession()` client-side. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-32-01 | Spoofing | /settings hub page | mitigate | Rely on `proxy.ts` session middleware that already protects `/settings/*`. No new auth surface introduced. |
| T-32-02 | Tampering | /profile redirect | mitigate | Server-side `redirect()` (Next.js navigation) cannot be intercepted by the client; redirect target is hardcoded to `APP_ROUTES.profileSettings`. |
| T-32-03 | Information Disclosure | components/settings/settings-hub.tsx | accept | Component renders static link copy only; no user data. Safe to be a server component without `verifySession()`. |
| T-32-04 | Elevation of Privilege | components/layout/topbar.tsx | mitigate | Link href change only; the topbar already runs through `authClient.useSession()` and the destination is still gated by `proxy.ts`. |
</threat_model>
