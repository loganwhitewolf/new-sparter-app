<objective>
Fix the gap surfaced by 64-UAT.md Test 1 / .planning/debug/64-smart-back-filter-loss.md: Plan
64-05's smart-back `router.back()` path correctly restores the browser history entry (URL query
string intact), but Next.js 16's App Router Client Cache unconditionally reuses the destination
table's previously-rendered RSC payload on back/forward navigation — independent of
`staleTimes.dynamic` — so the table can visually render a stale pre-filter snapshot even though
the address bar and `useSearchParams()`-derived toolbar chips correctly show the restored
filtered URL. Confirmed root cause (not re-diagnosed here): Next's back/forward Client Cache
reuse is documented as unconditional ("Pages are not cached by default but are reused during
browser back/forward navigation" — `staleTimes.dynamic=0` does not change this), and the
transactions/expenses/import tables write filter/sort state via `router.replace()`
(`components/data-table/use-table-url.ts`), which updates the current history entry in place.

Purpose: close this gap the same way Plan 64-05 fixed the original D-08 gap — once, in the one
shared `DetailPageShell`, so all three detail pages (transaction, expense, file) inherit the fix
without per-page changes. Fix is scoped to the exact call site the debug session traced
(`router.back()` inside `handleBackClick`); `use-table-url.ts`'s `router.replace()` writes are
left untouched (switching those to `push()` was explicitly flagged in the debug session as a
worse tradeoff — it would spam browser history on every filter change).

Output: `DetailPageShell` pairs `router.back()` with a one-time `popstate` listener that calls
`router.refresh()`, busting the Client Cache for the destination route exactly once per
back-navigation, without adding a new history entry or losing scroll position.
</objective>
