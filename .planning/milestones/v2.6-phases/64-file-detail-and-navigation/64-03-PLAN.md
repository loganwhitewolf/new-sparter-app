<objective>
Build the `/import/[fileId]` detail page (DET-08): an RSC route that ownership-checks and
status-branches the file, plus a client component that renders it inside the shared
`DetailPageShell` with `displayName` editable inline, platform/format/stats readonly, a capped
transactions preview list (each row linking out), and the existing file actions (download,
suggestions, delete) lifted into the page header.

Purpose: this is the third and final detail page in the v2.5 trilogy (`/transactions/[id]`,
`/expenses/[id]` shipped in Phase 63) — the file becomes a first-class navigable entity instead
of only being reachable as a table row or a `?fileId=` filter target.

Output: new route `app/(app)/import/[fileId]/page.tsx`, new client component
`components/import/file-detail-client.tsx`, DAL/RSC test coverage.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→RSC route param | fileId arrives from the URL path segment, unauthenticated until verifySession() + getFileDetailForUser() run |
| client→server action | delete/recheck/rename actions all re-verify session + ownership server-side, never trusting the fileId alone |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-64-04 | Tampering | FileDetailPage status redirect | medium | mitigate | Status branching happens via server-side `redirect()`/`notFound()` in the RSC — cannot be bypassed by client navigation or a modified URL; the destination pages (analyze/configure) re-verify ownership independently |
| T-64-05 | Information Disclosure | FileDetailPage ownership check | high | mitigate | `getFileDetailForUser` scopes by `eq(file.userId, userId)` (Plan 64-01); a non-owned or non-existent fileId both resolve to the same `notFound()` outcome — no distinguishing signal between "doesn't exist" and "not yours" |
| T-64-06 | Elevation of Privilege | ImportDeleteDialog reused with a constructed row object | medium | mitigate | The constructed object only pads `platformId`/`platformSlug` to `null` for type compatibility; the actual delete goes through `deleteImportAction`, which re-verifies ownership server-side from `verifySession()` — the client-constructed row shape cannot grant delete rights it wouldn't otherwise have |

</threat_model>
