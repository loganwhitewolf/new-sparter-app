<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Client-supplied `transactionIds`/`tagIds` arrays → assignment service | The arrays are just JSON strings from FormData — the service re-verifies ownership of EVERY id in both arrays against the caller's `userId` before any write; a single forged id anywhere in either array rejects the whole call (fail-closed, not partial-write). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-67-09 | Tampering (IDOR) | `bulkAssignTags`/`bulkRemoveTags` | high | mitigate | `assertOwnsAllTransactions`/`assertOwnsAllTags` re-query both id sets against the caller's `userId` before any write; a request mixing in even one id owned by another user rejects entirely — no partial cross-user write is possible. |
| T-67-10 | Integrity (race) | `transaction_tag_tagId_transactionId_unique` + `onConflictDoNothing` | medium | mitigate | Two concurrent bulk-assigns for the same pair never create duplicate join rows — the DB constraint plus `onConflictDoNothing` (Plan 67-01 + this plan's Task 1) removes the need for a pre-check-then-insert race window. |
| T-67-11 | Tampering | `BulkAssignTagsSchema`/`BulkRemoveTagsSchema` array bounds | low | mitigate | `min(1).max(500)` on `transactionIds`, positive-int `tagIds` — malformed or oversized payloads are rejected before reaching the service. |
</threat_model>
