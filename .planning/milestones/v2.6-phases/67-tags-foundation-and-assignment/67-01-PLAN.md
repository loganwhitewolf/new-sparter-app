<objective>
Lay the additive schema foundation for Transaction Tags: two new tables (`tag`, `transactionTag`)
with a DB-level case/whitespace-insensitive uniqueness guard and a DB-level dedup guard on
tag-transaction assignment, applied via a real migration, plus the Zod validation schemas every
later DAL/service/action in this phase will import.

Purpose: every later plan (DAL, services, actions, UI) depends on these tables existing in the
live dev database and on the tag Zod schemas being importable. Getting the two DB-level unique
constraints right here ã `(userId, normalizedName)` on `tag` and `(tagId, transactionId)` on
`transactionTag` ã is what actually closes the TAG-01 concurrency edge (two racing duplicate-name
creates) and the TAG-02 duplicate-assignment race, not merely a service-layer pre-check.

Output: `lib/db/schema.ts` additions, a generated+applied SQL migration, `lib/validations/tags.ts`,
one exported helper promoted from `lib/validations/transactions.ts`, and one new route constant.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Client form data ã tag CRUD / bulk-assign actions (built in Plans 67-03/67-04) | `CreateTagSchema`/`UpdateTagSchema`/`BulkAssignTagsSchema`/etc. are the first parse boundary for untrusted name/date-range/id-array input before any DB write. |
| Concurrent duplicate-name create | Two racing requests for the same normalized tag name ã the service-level pre-check alone (built in Plan 67-03) cannot close this; it needs a DB constraint. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-67-01 | Tampering | `tag_userId_normalizedName_unique` | high | mitigate | DB-level unique constraint on `(userId, normalizedName)` closes the TOCTOU race a service-only pre-check would leave open (TAG-01 concurrency edge) ã the second concurrent insert always fails with Postgres error 23505, caught and mapped by Plan 67-03. |
| T-67-02 | Tampering | `CreateTagSchema`/`BulkAssignTagsSchema` | medium | mitigate | Array bounds (`min`/`max`) plus per-item `uuid()`/positive-int format reject malformed or oversized payloads before they reach the service layer built in later plans. |
| T-67-03 | Repudiation / Integrity | `transaction_tag_tagId_transactionId_unique` | medium | mitigate | DB-level unique constraint lets Plan 67-04's bulk-assign use `onConflictDoNothing`, guaranteeing a concurrent double bulk-assign never creates duplicate join rows. |
| T-67-SC | Tampering | npm/pip/cargo installs | n/a | accept | Phase 67 adds no new package (RESEARCH.md Package Legitimacy Audit confirms); Package Legitimacy Gate is not applicable this plan. |
</threat_model>
