  <name>Task 1: Add per-category breakdown to getTagDetail / buildTagDetailData</name>
  <files>lib/dal/tags.ts, tests/tags-dal.test.ts</files>
  <behavior>
    - buildTagDetailData over rows with mixed category names groups into one entry per category, each with the signed Decimal sum of that category's amounts (toFixed(2)) and the row count.
    - breakdown is sorted by absolute total descending (mirror buildTagTotalsData's Decimal .abs().comparedTo sort).
    - Σ breakdown.total (Decimal) === detail.net, and Σ breakdown.count === detail.count === transactions.length (reconciliation invariant, TAG-07/TAG-08).
    - An allocation-style row (contributes to net but neither in nor out) still lands in its category's breakdown bucket, so the invariant holds even when inflow − outflow ≠ net.
    - Empty rows → breakdown: [] (and net "0.00", count 0).
  </behavior>
  <action>
Extend the EXISTING query and pure shaper in lib/dal/tags.ts — do NOT add a second query (D3).

1. In the `getTagDetail` select object, add `categoryName: category.name` (category is ALREADY innerJoined on line ~335, so this adds a column, never a row — the netted row set and therefore `net` are unchanged, preserving reconciliation with getTagTotals, TAG-07). Add `categoryName: string` to the `TagDetailQueryRow` type.

2. Add an exported `TagBreakdownItem = { categoryName: string; total: string; count: number }` type (signed DECIMAL string total, per the D3 shape). Add `breakdown: TagBreakdownItem[]` to the `TagDetail` type with a comment noting total is signed.

3. In `buildTagDetailData`, while iterating rows accumulate a per-category map keyed by `categoryName` (a Decimal running total + integer count), using toDecimal / Decimal.js `.plus` for the sum — never native arithmetic (CLAUDE.md). After the loop, materialize the map to `TagBreakdownItem[]` (`total: acc.toFixed(2)`), then sort by absolute total descending using `toDecimal(b.total).abs().comparedTo(toDecimal(a.total).abs())` (same rule as buildTagTotalsData). Return it as `breakdown` on the existing result object; leave inflow/outflow/net/count/transactions exactly as they are.

Add a `describe('buildTagDetailData (pure)')` block covering the behaviors above — including an explicit reconciliation assertion that the Decimal sum of breakdown totals equals net and the sum of breakdown counts equals count equals transactions.length, and the empty-rows case. Follow the file's existing pure-function test style (buildTagTotalsData block); no DB, no new mock wiring.
