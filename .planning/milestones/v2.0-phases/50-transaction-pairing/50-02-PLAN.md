<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| service → DB (transaction_pair writes) | The pair table has no userId column (D-01); ownership is enforced one tier up in the Plan 03 service. The DB enforces only FK integrity + uniqueness. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-50-01 | Information Disclosure / Tampering (IDOR) | `transaction_pair` has no userId | mitigate | Schema deliberately omits userId so ownership cannot be spoofed by a forged column; both FKs cascade from `transaction`, which carries userId. The actual ownership check lands in Plan 03's `createPair`/`deletePairByTransactionId`. Acceptance: schema has no userId column. |
| T-50-02 | Tampering (double-link integrity) | `transaction_pair_a_unique` + `transaction_pair_b_unique` | mitigate | Two single-column unique constraints guarantee a transaction can appear at most once as A and once as B — the DB-level safety net behind D-02. Acceptance: both unique constraints present in schema + 0020 SQL. |
| T-50-06 | Tampering (data integrity on delete) | FK `ON DELETE CASCADE` (D-03) | accept | Deleting either transaction auto-removes the pair row; the counterpart returns to unpaired with no orphan. This is the intended behavior, not a vulnerability. |
| T-50-SC | Tampering | npm/pip/cargo installs | accept | No package installs (50-RESEARCH.md §Package Legitimacy Audit: not applicable). |
</threat_model>
