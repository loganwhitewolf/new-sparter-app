<objective>
Build the targeted DB verification harness that proves the migrated database
satisfies the Phase 48 contract (D-04, MIG-03). This is the seventh step of the
canonical operator order (D-06) and the assertion set the staging-first gate
(D-14) must pass before production is touched.

The script reuses the existing operator connection plumbing (scripts/db-config.ts
+ the seed-extras runner shape) so it honours the same `.env` / `--staging` /
`--production` target resolution and production-confirm guard. It separates FATAL
assertions (active system subcategory without nature_id; residual pattern
duplicates) from INFORMATIONAL counts (override backfill coverage; allowed
user-owned null rows) per D-01/D-03/D-04.

Purpose: A migration that "ran without error" is not proof the data is correct —
this script is the objective gate for both staging sign-off and production apply.
Output: `scripts/verify-migration.ts`, `db:verify*` package scripts, and a pure
unit test of the pass/fail classifier.
</objective>

<decision_coverage>
This verification harness is the objective check that the seed/seed-extras data
transforms (authored in Phase 47, applied in Plan 03) produced a correct end state.
Several Phase 48 decisions are realised as assertions rather than new code here:

- D-08 (seed/seed-extras idempotent, tolerate reruns): the verification script is
  read-only and re-runnable; running it after a repeated db:seed-extras must yield
  the same PASS — it proves the idempotent steps converged, it does not mutate state.
- D-09 (recategorization target = expense.sub_category_id + categorization_pattern.sub_category_id;
  raw transaction rows stay intact): the script asserts categorization correctness on
  sub_category/categorization_pattern only and never inspects or mutates raw transaction
  rows — encoding the D-09 boundary that transactions are not the recategorization target.
- D-10 (slug-map driven, no description/sign/merchant inference): verification confirms
  the deterministic slug-map backfill (v2-backfill-nature-id) landed every active system
  subcategory's nature_id; no heuristic classification is performed or expected.
- D-11 (pattern merges follow migrateSubcategoryMerge: delete duplicate-target patterns,
  then move): the (pattern, sub_category_id) duplicate assertion proves the merge dedup
  ran correctly and that the sign-agnostic constraint swap left no residual collisions.
- D-12 (do NOT create expense_classification_history rows): no part of this plan (or the
  seed-extras transforms it verifies) writes expense_classification_history; the script
  asserts only nature_id/pattern state, consistent with "technical model migration, not a
  runtime categorization event."
</decision_coverage>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → target DB (staging/production) | verify-migration.ts connects to real deployed DBs using the same secrets as migrate/seed; a leaked connection string or unsanitized error would expose credentials. |
| verification result → go/no-go decision | A false PASS (assertion too lax) would let a broken migration reach production via the Plan 03 gate. |

| T-48-05 | Spoofing / wrong-target | operator target resolution | mitigate | Reuse resolveOperatorDatabaseTarget + getOperatorDatabaseConfig so production requires PRODUCTION_MIGRATION_CONFIRM; no bespoke env parsing. |

| T-48-SC | Tampering | npm/pip/cargo installs | mitigate | No new package installs (pg, drizzle-orm, tsx already present). If an install is needed, stop and add a blocking-human legitimacy checkpoint. |
