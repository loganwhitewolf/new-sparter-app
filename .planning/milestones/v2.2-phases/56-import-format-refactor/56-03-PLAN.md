<objective>
Migrate the live data and the seed source so the contract lives on `import_format_version`, then drop it from `platform` (D: ADR 0013 migration path; IFMT-03 prod-row migration, IFMT-05 seed half). Order inside this plan is load-bearing: (1) add the seed-extras data-copy STEP, (2) rework seed-data/seed.ts so the seeded contract is written on the version row, (3) ONLY THEN generate the migration that drops platform contract columns and tightens the import_format_version columns to NOT NULL. The seed-extras STEP is what guarantees existing production rows keep their contract across the drop (Claude's-Discretion: "the seed-extras data copy runs before any drop and the result is idempotent").

This plan still does NOT touch the runtime consumers (detector, DAL, type, wizard) — those move in Plan 04. After this plan the seed layer is correct but the detector DAL still reads the contract from the platform join; that join shape is updated in Plan 04. Keep that ordering so each plan is independently coherent.

Purpose: Make production and seeds own the contract on the version row; finalize the schema by dropping the now-redundant platform columns.
Output: idempotent seed-extras STEP, reworked seed scripts, platform-pure-identity schema, generated drop/tighten migration.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| seed-extras → production rows | An UPDATE over live platform/import_format_version rows; an incorrect or non-idempotent copy corrupts every import contract. |
| drop migration → production DB | Destructive DDL removing platform columns; if applied before the data copy, every platform loses its contract. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-56-04 | Tampering | seed-extras contract copy | high | mitigate | Idempotent guarded UPDATE (only where target contract is NULL); rowCount logged; copies all twelve columns verbatim so values are byte-identical to platform originals. |
| T-56-05 | Denial of Service | platform-column DROP applied before data copy | critical | mitigate | Operator run order is fixed (CLAUDE.md): db:migrate (add) → db:seed → db:seed-extras (copy) → db:migrate (drop); the drop migration is generated last and the SUMMARY documents the required apply order. |
| T-56-06 | Tampering | headerSignature drift after seed source move | high | mitigate | headerSignatureFor rekept to emit the identical five-column delimiter-joined string; Plan 04 re-runs the IFMT-02 regression test which would fail on any detection drift. |
</threat_model>
