# Requirements Archive: Sparter v1.10 Pattern Suggestions

**Archived:** 2026-05-25
**Original Defined:** 2026-05-22
**Status:** ✅ All 24 v1 requirements shipped

## v1 Requirements (all complete)

### Detection

- [x] **SUG-01**: User receives pattern suggestions for recurring valid uncategorized import rows that share a common normalized token prefix. *(Phase 33)*
- [x] **SUG-02**: Suggested patterns strip purely numeric tokens before prefix comparison. *(Phase 33)*
- [x] **SUG-03**: Suggested patterns require at least 2 matching rows and at least 2 non-numeric prefix tokens. *(Phase 33)*
- [x] **SUG-04**: Suggested patterns preserve the longest qualifying common prefix, not exactly 2 tokens. *(Phase 33)*
- [x] **SUG-05**: Suggested patterns exclude rows that are invalid, duplicate, or already matched by an active categorization pattern. *(Phase 33)*
- [x] **SUG-06**: Suggested regex sources are escaped so bank-description metacharacters cannot create unintended regex behavior. *(Phase 33)*

### Analysis Contract

- [x] **ANL-01**: Import analysis returns `patternSuggestions` in `ImportAnalysisResult`. *(Phase 34)*
- [x] **ANL-02**: Each pattern suggestion includes `pattern`, `matchCount`, `detectedAmountSign`, and up to 3 sample descriptions. *(Phase 33)*
- [x] **ANL-03**: Import analysis returns at most 5 pattern suggestions sorted by `matchCount` descending. *(Phase 34)*
- [x] **ANL-04**: `detectedAmountSign` is `positive`, `negative`, or `any` based on grouped transaction amounts. *(Phase 33)*
- [x] **ANL-05**: Pattern suggestion detection failures do not leak raw R2 object keys, presigned URLs, raw rows, or stack traces. *(Phase 34)*

### Import Review

- [x] **REV-01**: User can see pattern suggestions on the import analysis page before confirming import. *(Phase 35)*
- [x] **REV-02**: User can inspect sample descriptions for each suggestion. *(Phase 35)*
- [x] **REV-03**: User can select a destination subcategory and promote a suggestion to a categorization pattern. *(Phase 35)*
- [x] **REV-04**: User can continue import confirmation without handling suggestions. *(Phase 35)*
- [x] **REV-05**: User sees clear success or validation feedback after attempting to promote a suggestion. *(Phase 35)*

### Post-Import Re-Analysis

- [x] **POST-01**: User can re-run pattern suggestion analysis for an imported file using persisted transactions filtered by `fileId`. *(Phase 36)*
- [x] **POST-02**: Post-import re-analysis uses the same detection algorithm and suggestion shape as pre-import analysis. *(Phase 36)*
- [x] **POST-03**: Post-import re-analysis enforces session user ownership of the import file. *(Phase 36)*
- [x] **POST-04**: Post-import suggestions exclude transactions that already have category coverage. *(Phase 36)*
- [x] **POST-05**: User can promote a post-import suggestion to a categorization pattern. *(Phase 36)*

### Scope Boundaries

- [x] **SCOP-01**: Dismissed suggestions are not persisted. *(Phase 34 — by design, no dismissal table)*
- [x] **SCOP-02**: Pattern suggestions are scoped to one import file, not global transaction history. *(Phase 34)*
- [x] **SCOP-03**: Creating a post-import pattern does not automatically reclassify existing transactions unless a later requirement adds that behavior. *(Phase 36)*

## Future Requirements (deferred)

- **REVAL-01**: User can apply a newly created post-import pattern to existing transactions from the same import.
- **GLOBAL-01**: User can detect pattern suggestions across all uncategorized transaction history.
- **DISM-01**: User can permanently dismiss a noisy suggestion.

## Out of Scope

| Feature | Reason |
|---------|--------|
| ML/LLM clustering | ADR chose deterministic prefix detection |
| Longest common substring detection | Less anchored than prefix patterns |
| Persistent dismissal table | Explicitly excluded by ADR for v1.10 |
| Raw R2 post-import re-analysis | Persisted transactions are the source of truth after import |
| Automatic existing-transaction reclassification | Distinct from detection; deferred to REVAL-01 |
