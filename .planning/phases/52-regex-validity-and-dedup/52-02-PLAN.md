<objective>
Add the single new database read this phase requires: `getManuallyCategorizedHashes(userId, descriptionHashes)` in the DAL `lib/dal/regex-discovery.ts`. It returns the subset of the provided `descriptionHashes` that the user has ALREADY hand-categorized at least once, by querying `expenseClassificationHistory` with `source = 'manual'` joined to `expense.descriptionHash` (RDISC-04, Check 2 data source).

This query is the LOCKED data source for Check 2 (decision in 52-CONTEXT): it MUST query history, NOT current `expense` rows — the `unique(userId, descriptionHash)` constraint means an uncategorized group member and a categorized expense can never share a hash, so a current-state lookup returns nothing by construction. Manual classification history persists the user's decision independently of the live expense row. The query is platform-agnostic (descriptionHash is content addressing) and user-scoped (IDOR guard).

Purpose: Provide the server-only read that Plan 03's Check 2 wiring consumes — the only DB tier work in Phase 52.
Output: One new exported async DAL function returning `Set<string>`, mirroring the `applyTier2History` join shape, plus DAL unit tests asserting the WHERE clause structurally.
</objective>
