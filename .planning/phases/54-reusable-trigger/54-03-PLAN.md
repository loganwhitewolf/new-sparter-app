This plan (54-03) introduces / changes:

- **`recheckRegexAction(formData: FormData): Promise<ImportActionState<{ candidatesCount: number; singleCount: number; platformId: number }>>`** — new thin `"use server"` action in `lib/actions/import.ts`. Reads `fileId` from `formData`, resolves `userId` via `verifySession()`, resolves `platformId` via `getPlatformIdForUserFile({ userId, fileId })` (null → error), calls `discoverRegexCandidates({ userId, scope: { platformId } })`, returns the two counts + platformId. Read-only — no `revalidatePath`. Uses the existing `ImportActionState<T>` type already in `import.ts`.
- **`onRecheckRegex: (row: ImportListRow) => void`** prop + **`isRecheckPending?: boolean`** prop on `ImportRowActions` (`components/import/import-row-actions.tsx`), plus a new "ricontrolla regex" `DropdownMenuItem` (clickable, not a Link), shown for `row.status === 'imported'`.
- **Client wiring** in `components/import/import-table.tsx`: a `handleRecheckRegex(row)` that builds FormData, calls `recheckRegexAction`, then on `error` → `toast.error`, on zero total → neutral `toast('Nessun pattern trovato per questa piattaforma')` (no nav), on >0 → `router.push('/import/${fileId}/suggestions')`. Pending state passed to the row.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client (Files table) → recheckRegexAction | `fileId` is supplied by the client (the row id); it must not allow re-checking another user's file/platform |
| recheckRegexAction → DAL/service | `userId` is resolved server-side via `verifySession`; never accepted from the client |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-08 | Information Disclosure / Elevation | `recheckRegexAction` resolving `platformId` from a client-supplied `fileId` (IDOR) | mitigate | Resolve `platformId` ONLY via `getPlatformIdForUserFile({ userId, fileId })` using the verifySession-resolved userId; null → return an error before calling the service. A user cannot re-check another user's file (the guard scopes by userId). The suggestions page (54-01) re-validates ownership independently via `getPlatformIdForUserFile` + `notFound()`, so navigation cannot leak foreign data either. |
| T-54-09 | Spoofing | userId in `recheckRegexAction` | mitigate | `userId` comes from `await verifySession()`; the action never reads a userId from `formData` or any client input. |
| T-54-10 | Tampering | npm/pip/cargo installs | accept | No new package installs (reuses existing service, sonner, next/navigation). No legitimacy gate required. |
</threat_model>
