<objective>
Make the post-migration operator procedure an explicit, reviewable artifact and
actually drive the guarded apply. Because the lookup rows and all taxonomy/nature
data transforms live in seed/seed-extras (NOT in the generated SQL, D-05), the
correct sequence is not self-evident from the migration file alone — the runbook
captures it. The runbook also records the mandatory pre-migrate pg_dump and the
restore-from-dump rollback (D-13, the sole reversibility path since drops are
destructive and no down-migration exists), and codifies staging-first as a
blocking gate (D-14).

This plan then executes the guarded flow: rehearse on staging, run db:verify on
staging, STOP at a blocking human checkpoint, and only after approval apply +
verify production. The DB-apply commands are operator actions the user runs/approves
against real environments — Claude prepares, documents, and gates them; the human
authorises the production step.

Purpose: This is the step that physically moves the deployed database to v2.0,
safely and reversibly. It is the realisation of MIG-01/MIG-02/MIG-03 on live data.
Output: MIGRATION-RUNBOOK.md and a verified staging→production apply.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → production DB | The single most destructive boundary in the milestone: a bad apply on production can drop columns/enums irreversibly by SQL. |
| staging gate → production apply | The gate that must hold. Skipping staging verification risks shipping a broken migration to production. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-48-07 | Denial of Service / data loss | production db:migrate (destructive drops) | mitigate | Mandatory pre-migrate pg_dump (D-13) is step 3 of the runbook and a blocking checkpoint precondition; restore-from-dump is the documented recovery. |
| T-48-08 | Elevation / premature production apply | staging-first gate | mitigate | Production apply is a separate blocking-human checkpoint that cannot start until staging db:verify passes (D-14); the resume-signal explicitly gates it. |
| T-48-09 | Repudiation / unsafe rollback | no down-migration | accept | No reverse SQL exists by design; runbook mandates dump-based restore. Accepted with the dump as compensating control. |
| T-48-10 | Spoofing / wrong target | yarn db:*:production confirm | mitigate | PRODUCTION_MIGRATION_CONFIRM=apply-to-production gate (db-config.ts) plus the human checkpoint; no bespoke production path. |
| T-48-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs; pg_dump/pg_restore are system tools and yarn db:* scripts already exist. |
</threat_model>
