<objective>
Rework the three pattern-authoring surfaces per D-08/D-09/R-UP-07 and ADR 0008. The `create-pattern-dialog` becomes: a regex input + a description input + a "Categorizza" button that opens the unified `SubcategoryPicker`. The explicit Categoria/Sottocategoria `Select` pair, the "Segno importo" control, and the "Confidenza" input are all removed. The `suggestion-promote-form` likewise drops its cascading Selects in favor of the picker (keeping its pre-filled regex). The **edit-existing-pattern dialog** (`components/patterns/pattern-actions.tsx`, rendered in Settings â€ş Categories â€” adoption surface #8 in D-08) gets the SAME treatment: its cascading Categoria/Sottocategoria `Select` pair, its "Segno importo" `Select`, and its "Confidenza" `Input` are all removed and replaced with regex + description + a "Categorizza" picker button. `amountSign` is no longer a user field: it is derived **server-side** from the chosen subcategory's category type (`out -> negative`, `in -> positive`, `transfer`/`system -> any`) for `createPatternAction`, `promoteSuggestionAction`, AND `updatePatternAction`; `confidence` is hardcoded to `1` for all three. The `amountSignMatches` matching logic stays untouched.

Per A1 (non-breaking schema): the `amountSign`/`confidence` Zod fields stay on `CreatePatternSchema` (so `UpdatePatternSchema = CreatePatternSchema.partial()` and `updatePatternAction`'s parse keep working). The change is behavioral â€” the actions stop *reading* user-supplied `amountSign`/`confidence` and derive/hardcode them instead. No Zod field is deleted.

Purpose: Align all pattern authoring (create + promote + edit) with the unified picker and ADR 0008 (amountSign is a property of the subcategory's category, not a separate user choice). Close the duplicate-UX gap where the edit dialog still exposed a cascading Select pair and a user amountSign/confidence control.
Output: All three pattern surfaces use the picker; create/promote/update actions derive amountSign server-side and write confidence=1; the schema stays non-breaking; a unit test pins the derivation table.
</objective>
