<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `loadActivePatterns` / `detectPatternSuggestions` → caller | An error thrown inside these functions may carry an `error.message` containing URLs (R2 endpoints), stack frames (filesystem paths), or substrings derived from row data. The catch block in `analyzeFile` is the trust boundary that must sanitize before logging. |
| `logger.warn` payload → log sink | Anything passed to `logger.warn` is written to the structured log stream and may be persisted by the platform log collector. Sensitive fields in the payload object would leak there. |
| `ImportAnalysisResult` → action → page | The result is serialized over the React Server Components boundary. `PatternSuggestion.sampleDescriptions` already contains user-supplied bank descriptions — same trust posture as the existing `sampleRows.description` field, no new boundary. |

Pass ONLY `{ event, message, userId, fileId }` to `logger.warn` — never the raw `error`, the `normalizedRows`, the `objectKey`, or the rows array. Pinned by Wave 0 test case 4 in `tests/import-service.test.ts` ("returns patternSuggestions [] and logs a warning when detection throws") which asserts the input URL and stack-frame substrings do not appear anywhere in `mocks.loggerWarn.mock.calls`. | T-34-02 | Denial of Service | Cascading failure from suggestion catch to `markFileFailed` | MEDIUM | mitigate | The suggestion try/catch MUST NOT call `markFileFailed` and MUST NOT rethrow — detection is non-critical and the file must stay `analyzed`.
