<objective>
Wire `?tag=` end-to-end through the three dashboard pages that consume Plans 68-01/02/03/05's
work: read the searchParam, validate ownership via `resolveOwnedTagId`, forward the resulting
`tagId` to every DAL call on that page, render `TagFilterSelect`, and make sure the tag filter
survives navigation between the category ranking list and its detail sub-route (which currently
only forwards `preset`/`type`/`sort`).

Purpose: This is the page-level half of TAG-04 — Plans 68-02/03 made the DAL layer capable of
narrowing by tag; this plan is what actually reads the user's selection and threads it through on
every one of the three routes, closing "every existing widget narrows."

Output: Overview tab, Categorie tab, and category detail page all correctly narrow by tag, with
the filter surviving tab switches (Plan 68-05) and ranking-list-to-detail navigation.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Browser -> `?tag=` on all three dashboard pages | User-controlled value; this is the primary entry point where an attacker could attempt to reference another user's tagId |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-68-01 | Tampering / Information Disclosure | `?tag=` on `/dashboard/overview`, `/dashboard/categories`, `/dashboard/categories/[id]` | medium | mitigate | Every one of the three pages calls `resolveOwnedTagId(userId, parseTagIdParam(params))` (Plan 68-01) BEFORE forwarding any value to a DAL call — a foreign or malformed `tagId` resolves to `undefined` and is silently ignored (fail-closed), never thrown as an error and never reaching a query. This is the concrete, page-level closure of the threat identified during planning: the explicit ownership check recommended as defense-in-depth on top of the already-safe transitive `userId` scoping in every touched query (Plans 68-02/68-03). |
| T-68-04 | Tampering | `buildDashboardCategoryDetailHref`/`buildDashboardCategoriesHref`'s new `tag` param | low | mitigate | The `tag` value forwarded through these href builders always originates from an already-`resolveOwnedTagId`-validated `tagId` in scope on the rendering page — never re-derived from raw, unvalidated user input at the link-construction site. |

</threat_model>
