<objective>
Implement `updateTransaction` — a backend service (+ thin server action) that lets a user edit a transaction's `amount`, `occurredAt`, and `customTitle` inside a single `db.transaction`, while `transactionHash`, `descriptionHash`, and `description` stay frozen. An amount or date edit atomically reconciles the linked expense's derived aggregates using the existing `expense-reconciliation` service. An amount edit on a paired transaction that would break the pair's opposite-sign/nonzero invariant is rejected before any write, with the Italian message "Scollega prima il rimborso".

Purpose: This is the backend foundation Phase 63's detail pages will call for pencil-inline editing — no UI in this phase.
Output: `lib/services/transaction-edit.ts` (service), `lib/validations/transaction-edit.ts` (Zod schema), `lib/actions/transaction-edit.ts` (thin `"use server"` action), `tests/transaction-edit.test.ts` (RED→GREEN coverage for DET-01/02/03).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Client (form/action call) → `updateTransactionAction` | Untrusted `FormData` (id, amount, occurredAt, customTitle) crosses into the server action. |
| `updateTransactionAction` → `updateTransaction` service | `userId` is derived server-side via `verifySession()`, never taken from the client payload. |
| `updateTransaction` service → DB | Ownership-gated reads/writes on `transaction`; `transactionPair` has no `userId` column (documented risk from Phase 50). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-62-01 | Tampering | `updateTransaction` — IDOR on transactionId | high | mitigate | Load with `where(and(eq(transaction.id, ...), eq(transaction.userId, input.userId)))`; throw `'Transazione non trovata.'` when the row is absent or owned by another user — the two cases are indistinguishable to the caller by design (no user enumeration). |
| T-62-02 | Tampering | `updateTransaction` — hash/description injection via extra input keys | high | mitigate | Service-level `UpdateTransactionInput` type has no `transactionHash`/`descriptionHash`/`description` fields; the `.set()` payload is built key-by-key from an explicit allowlist (amount/occurredAt/customTitle/updatedAt only), so no code path can assign those columns regardless of caller input. |
| T-62-03 | Tampering | `updateTransaction` — pair invariant bypass | high | mitigate | Pair-guard pre-check runs and can throw before any `tx.update` call; Task 1 test 7 asserts `tx.update(transaction)` is never invoked when the guard rejects. |
| T-62-04 | Tampering | Reconciliation race — edit and reconcile split across requests | high | mitigate | Both the transaction UPDATE and the reconciliation calls run inside one `db.transaction(async (tx) => ...)`, with `tx` (not `db`) passed to every reconciliation helper. |
| T-62-05 | Information Disclosure | `transactionPair` counterpart read leaks another user's amount | medium | accept | The counterpart transaction is only readable because it is already linked via `transactionPair` to a transaction the caller owns; Phase 50's `createPair` already established this trust model (pair creation itself is ownership-gated on both legs), so no new exposure is introduced by reading the counterpart's amount for the invariant check. |
| T-62-06 | Repudiation | No audit trail for who changed amount/date/title | low | accept | Out of scope per REQUIREMENTS.md ("Out of scope: ... revision history/audit log"); `updatedAt` timestamp is the only trace, consistent with existing edit paths (`updateTransactionCustomTitle`). |

</threat_model>
