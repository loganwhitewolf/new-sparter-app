remove member with auto-dissolve, dissolve whole group) needed by Phase 66's lifecycle
operations (GRP-05 recategorize, GRP-06 add-to-group, GRP-07 remove/dissolve). This plan is

each of the three new mutations, so the actions layer stays a thin orchestration wrapper
(parse → verifySession → db.transaction → call service → revalidate), matching the repo's
established action/service split.

Output: 4 new Zod schemas in `lib/validations/expense.ts`; 3 new exported functions in
`lib/services/expense-group.ts`; extended `tests/expense-group-service.test.ts` coverage.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| server action (Plan 66-02) → service (this plan) | userId comes from verifySession() upstream; groupId/expenseId(s) come from user-controlled FormData — this plan's functions are the last IDOR gate before any DB write. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-66-01 | Tampering/Elevation | addExpensesToGroup | medium | mitigate | Every mutation path re-verifies `expenseGroup.userId = userId` AND `expense.userId = userId` inside the same DbOrTx call before any insert — a caller cannot add another user's expense to their own group, nor add an expense to another user's group, even if the calling action's own checks were somehow bypassed (defense in depth, mirrors createExpenseGroup). |
| T-66-02 | Tampering | addExpensesToGroup membership insert | medium | mitigate | Concurrent add-to-group races on `unique(expenseId)` are caught via `errorCauseCode(e) === '23505'` and translated to the same Italian message as the pre-check — no raw Postgres error/stack ever reaches the client. |
| T-66-03 | Tampering (TOCTOU) | removeExpenseFromGroup auto-dissolve | medium | mitigate | The membership count is read and the auto-dissolve decision made INSIDE the same DbOrTx call/transaction as the delete — Plan 66-02 must call this from within `db.transaction`, so no other request can insert/remove a membership row between the count and the delete. |
| T-66-04 | Information Disclosure | removeExpenseFromGroup / dissolveExpenseGroup | low | mitigate | All not-found/not-owned paths return the identical generic 'Gruppo non trovato.' / 'Spesa non trovata nel gruppo.' message regardless of whether the id truly doesn't exist or simply isn't owned by userId — no distinction leaks group existence to a non-owner. |
</threat_model>
