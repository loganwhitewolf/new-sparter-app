<objective>
Add the missing UI control that lets a user filter the transactions table by tag from the toolbar, integrated into the existing unified declarative filter/sort system (chip, clear-all, sessionStorage persistence) — satisfying TAG-14.

Everything downstream of the `?tag=` URL param already exists and is correct (LOCKED per CONTEXT): `parseTransactionFilters` reads `input.tag` into `ParsedTransactionFilters.tagId`; `page.tsx` runs `resolveOwnedTagId` (fail-closed IDOR guard) and passes `tagId` to `getTransactions`, which applies `tagScopedTransactions(tagId)`. This plan adds ONLY the toolbar control that writes `?tag=`; it changes none of that plumbing (scope fences: no change to `getTransactions`, `resolveOwnedTagId`, or `?tag=` parsing).

The transactions filter system is declarative: `transactionsTableConfig.filters` (D1) drives `DataTableToolbar`, which gives active-chip + "Cancella tutto" + persistence automatically for any entry. The toolbar's `key -> searchParam` mapping is 1:1 (it reads/writes `searchParams.get(field.key)` and clears via `field.key`), so a `{ key: 'tag' }` entry binds directly to the existing `?tag=` param (D3) — no new param.

One design fact drives the shape of the work: the chip builder today passes only the raw URL value to `field.toChip`. The tag value is `String(tag.id)` (D2, required by the numeric `?tag=` DAL path), but the chip must show the tag NAME (D1). Because the config is a static module, the name cannot be closed over there. The fix is a backward-compatible widening of `toChip` to receive the resolved option label as an optional 2nd argument, which the tag filter uses and every existing filter ignores.

Do NOT reuse or touch `TagFilterSelect` — that is the dashboard control removed in Phase 70; this phase is independent (D4).

Purpose: give the transactions list a tag *navigation* filter (complements TAG-13's removal of the dashboard tag *analysis* filter).
Output: a working tag control in the transactions toolbar, wired end-to-end to the existing param + DAL, with a name-bearing chip, clear-all, and persistence.
</objective>
