<objective>
Build the read-time composition layer: `getExpenses` renders one row per Expense Group instead of
N member rows (with Decimal.js-summed totals), `getExpenseGroupForDetail` powers the group detail
page, and `transactionListSelect`/`getTransactionForDetail` learn the group-title display
precedence — all with zero behavior change for anything that isn't grouped.

Purpose: this is the only plan that touches the read/query layer; every UI plan in the next wave
consumes the types and functions built here. The composition algorithm (collapse-then-paginate,
never paginate-then-collapse) is the single highest-risk piece of this phase per RESEARCH.md's
pitfall analysis, so it gets its own dedicated, test-first task.
Output: extended `lib/dal/expenses.ts`, extended `lib/dal/transactions.ts`, one new route helper.
</objective>
