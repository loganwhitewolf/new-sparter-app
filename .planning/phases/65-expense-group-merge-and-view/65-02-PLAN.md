<objective>
Implement the write side of the Expense Group model: a service layer that creates and renames
groups with ownership + one-group-per-expense enforcement, and the server actions that expose
that service to the client — while keeping the merge action structurally incapable of assigning
a category (D-02), and closing a D-03 bypass on the pre-existing single-expense categorize action.

Purpose: this is the only place in the phase where expenseGroup/expenseGroupMembership rows are
written. Every UI plan later in this phase calls into `mergeExpenses`/`renameExpenseGroupAction`
built here; getting the ownership and category-gate checks right here is what makes the rest of
the phase safe to build on top of.
Output: `lib/services/expense-group.ts`, extended `lib/actions/expenses.ts`, and their tests.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Client formData → mergeExpenses / renameExpenseGroupAction / categorizeExpense | All three are server actions taking client-supplied ids; each must independently re-verify ownership server-side (no client-side check is trustworthy). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-65-03 | Elevation of Privilege / Tampering | mergeExpenses | high | mitigate | Ownership-scoped SELECT (`userId` AND `id IN (deduped ids)`) runs before any write; a row-count mismatch against the deduped input rejects the whole request — prevents merging another user's expense by id-guessing (IDOR). |
| T-65-04 | Information Disclosure / Tampering | renameExpenseGroupAction | high | mitigate | `renameExpenseGroup`'s UPDATE is scoped to `expenseGroup.userId = userId`; zero affected rows is treated as a thrown error, never a silent no-op that could mask a cross-user rename attempt. |
| T-65-05 | Tampering | categorizeExpense (existing action, now guarded) | medium | mitigate | The new `expenseGroupMembership` pre-check blocks recategorizing a grouped member directly, closing the D-03 bypass a forged request to this pre-existing single-expense action could otherwise use. |
| T-65-06 | Tampering | bulkCategorize (unchanged this plan) | low | accept | Grouped members are never rendered as selectable rows in the expenses table (enforced in Plan 65-04), so bulkCategorize is unreachable for a grouped id through the UI; a forged direct request would only violate the D-03 UI invariant on the caller's own data (no cross-user exposure), so it is left unguarded this phase. |
</threat_model>
