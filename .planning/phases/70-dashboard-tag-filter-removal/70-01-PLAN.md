<objective>
Remove the period-scoped `?tag=` dashboard filter from every SURFACE that reads or renders it:
the three dashboard pages (`/dashboard/overview`, `/dashboard/categories`,
`/dashboard/categories/[id]`), the two dashboard href builders, the category ranking list's
tag-conditional copy, the overview empty-state variant, and the tag argument chain that runs from
the overview page down through the movers section into the `fetchMovers` Server Action.

This plan is a PURE REMOVAL. No behaviour is substituted and no new affordance is added (D2).
Per-tag analysis already lives in `/tags/[id]` (Phase 69) and is reached from `/dashboard/tags`.

The dashboard DAL keeps its optional `tagId` parameters for now — nobody passes them after this
plan, and plan 70-02 deletes the parameters themselves. Splitting it this way keeps `tsc --noEmit`
clean after every single task.

**REGRESSION FENCE (LOCKED, do not cross):** `lib/dal/transaction-tags-sql.ts` and its
`tagScopedTransactions` export, `lib/dal/transactions.ts`, `resolveOwnedTagId` in
`lib/dal/tags.ts`, `app/(app)/transactions/**`, and the Phase 71 transactions toolbar tag filter
(TAG-14, just shipped) are OUT OF SCOPE. `/transactions?tag=` must keep working. A change that
deletes the shared helper or the transactions call site is WRONG.

Purpose: a tag must show ONE reconciled set of numbers. Crossing a tag with the dashboard's
year/month scope produced a second, divergent set — that is the inconsistency being removed.
Output: three dashboard pages and their component/action chain with zero tag wiring.
</objective>
