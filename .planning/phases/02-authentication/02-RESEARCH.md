### Locked Decisions

- **D-01:** `/register` form collects only email + password — no name fields at signup. firstName/lastName deferred to Phase 7.
- **D-02:** Post-registration redirect goes directly to `/dashboard` (auto-login via Better Auth `autoSignIn: true`).

- **D-06:** Email already registered message: "Si è verificato un errore. Riprova." — generic.

- **D-11:** Password minimum: 8 characters (Zod v4 `z.string().min(8)`). No complexity requirements.

- **D-13:** Better Auth uses Drizzle adapter with PostgreSQL (`drizzle-orm/node-postgres`) and provider `pg`. Tables generated via schema generation + `drizzle-kit generate` + `npm run db:migrate`.
- **D-14:** `lib/db/index.ts` stub replaced with real `pg` Pool connection in Phase 2.

## Summary

Phase 2 installs and wires Better Auth v1.6.9 — the project's locked auth provider — with the Drizzle PostgreSQL adapter. The phase connects the existing `lib/db/index.ts` stub to a real `pg` Pool, generates and migrates the Better Auth schema (users, sessions, accounts, verifications tables plus custom `subscriptionPlan` and `role` columns), configures `auth.ts` at the project root, creates the `/api/auth/[...all]` route handler, and wires the login/register form pages (currently static stubs) to server actions backed by `auth.api.signInEmail` / `auth.api.signUpEmail`.

Route protection moves from the Phase 1 pass-through `proxy.ts` to a real session check via `auth.api.getSession`. The staging bypass (`x-staging-key`) is checked first, before the auth check, and activates whenever `STAGING_KEY` env var is set. A `verifySession()` helper in `lib/dal/auth.ts` provides the authoritative session check for all downstream phases. Topbar is wired to show live session data.

**Primary recommendation:** Use the Better Auth server-side API (`auth.api.signInEmail`, `auth.api.signUpEmail`) from Server Actions with the `nextCookies()` plugin for automatic cookie setting. Use `auth.api.getSession({ headers: await headers() })` for all server-side session reads. Do NOT use the Better Auth client SDK for server-side auth flows.

### Pattern 1: Better Auth Configuration (`auth.ts`)

**What:** Central auth config at project root. Exports `auth` instance.
**When to use:** Single instance — imported everywhere auth is needed.

```typescript
// Source: https://better-auth.com/docs/installation + https://better-auth.com/docs/concepts/database
// auth.ts (project root)
import { betterAuth } from "better-auth"
import { drizzleAdapter } from "better-auth/adapters/drizzle"
import { nextCookies } from "better-auth/next-js"
import { db } from "@/lib/db"

export const auth = betterAuth({
  secret: process.env.BETTER_AUTH_SECRET!,
  baseURL: process.env.BETTER_AUTH_URL,
  emailAndPassword: {
    enabled: true,
    minPasswordLength: 8,
    autoSignIn: true,  // D-02: auto-login after signup → redirect to /dashboard
  },
  database: drizzleAdapter(db, {
    provider: "pg",
  }),
  user: {
    additionalFields: {
      subscriptionPlan: {
        type: ["free", "basic", "pro"],  // D-09
        required: false,
        defaultValue: "free",
        input: false,  // users cannot set this at signup
      },
      role: {
        type: ["user", "admin"],  // D-09
        required: false,
        defaultValue: "user",
        input: false,  // users cannot set this at signup
      },
    },
  },
  plugins: [nextCookies()],  // MUST be last plugin — auto-sets cookies in Server Actions
})
```

### Pattern 3: Server Actions for Sign-in / Sign-up

**What:** Thin "use server" wrappers that validate with Zod v4, call `auth.api.*`, handle errors.
**When to use:** Form submission from login/register Client Components via `useActionState`.

**Critical notes:**
- `auth.api.signUpEmail` requires a `name` field — Better Auth does NOT allow omitting it. Workaround: pass `email` value as `name` (or a derived placeholder like the email prefix). [VERIFIED: github.com/better-auth/better-auth/issues/7120]
- `auth.api.signInEmail` / `auth.api.signUpEmail` throw an `APIError` on failure — wrap in try/catch and check `error.message`.
- The `nextCookies()` plugin handles cookie propagation automatically; no manual `cookies().set()` needed.
- After `auth.api.signUpEmail` with `autoSignIn: true`, the session cookie is set; then call `redirect('/dashboard')`.

```typescript
// Source: https://better-auth.com/docs/integrations/next (server action cookies section)
// lib/actions/auth.ts
'use server'
import { auth } from "@/auth"
import { headers } from "next/headers"
import { redirect } from "next/navigation"
import { LoginSchema, RegisterSchema } from "@/lib/validations/auth"

export type AuthActionState = { error: string | null }

export async function signInAction(
  _prev: AuthActionState,
  formData: FormData
): Promise<AuthActionState> {
  const parsed = LoginSchema.safeParse({
    email: formData.get("email"),
    password: formData.get("password"),
  })
  if (!parsed.success) {
    return { error: "Credenziali non valide. Riprova o contatta il supporto." }
  }
  try {
    await auth.api.signInEmail({
      body: { email: parsed.data.email, password: parsed.data.password },
      headers: await headers(),
    })
  } catch {
    // D-05: generic message — do not reveal whether email exists
    return { error: "Credenziali non valide. Riprova o contatta il supporto." }
  }
  redirect("/dashboard")
}

export async function signUpAction(
  _prev: AuthActionState,
  formData: FormData
): Promise<AuthActionState> {
  const parsed = RegisterSchema.safeParse({
    email: formData.get("email"),
    password: formData.get("password"),
  })
  if (!parsed.success) {
    return { error: "Si è verificato un errore. Riprova." }
  }
  try {
    await auth.api.signUpEmail({
      body: {
        email: parsed.data.email,
        password: parsed.data.password,
        name: parsed.data.email,  // Required by Better Auth — use email as placeholder
      },
      headers: await headers(),
    })
  } catch {
    // D-06: generic message — do not reveal email already exists
    return { error: "Si è verificato un errore. Riprova." }
  }
  redirect("/dashboard")
}

export async function signOutAction(): Promise<void> {
  await auth.api.signOut({ headers: await headers() })
  redirect("/login")
}
```

### Pattern 4: Zod v4 Validation Schemas

**What:** Validation schemas using Zod v4 API (breaking changes from v3).
**When to use:** Client-side fast-fail + server-side authoritative validation.

**Key Zod v4 breaking changes affecting Phase 2:** [VERIFIED: zod.dev/v4/changelog]
- `z.string().email()` → `z.email()` (top-level function, no longer a string method)
- `z.string().min(8, { message: "..." })` → `z.string().min(8, { error: "..." })` (key renamed)
- `.flatten()` still works in Zod v4 but `.format()` is deprecated
- `z.nativeEnum()` deprecated — use `z.enum()` for everything
- `safeParse()` API unchanged — still returns `{ success: boolean, data, error }`

```typescript
// Source: https://zod.dev/v4/changelog
// lib/validations/auth.ts
import { z } from "zod"

export const LoginSchema = z.object({
  email: z.email({ error: "Email non valida." }).trim(),
  password: z.string().min(8, { error: "Password troppo corta." }),
})

export const RegisterSchema = z.object({
  email: z.email({ error: "Email non valida." }).trim(),
  password: z.string().min(8, { error: "La password deve essere di almeno 8 caratteri." }),
})

export type LoginInput = z.infer<typeof LoginSchema>
export type RegisterInput = z.infer<typeof RegisterSchema>
```

### Pattern 5: proxy.ts — Route Protection with Staging Bypass

**What:** Next.js 16 `proxy.ts` with Better Auth session check and staging bypass.
**When to use:** Route-level redirect (optimistic guard). NOT the last line of defense — DAL `verifySession()` is authoritative.

**Critical notes:**
- proxy.ts runs in Node.js runtime by default in Next.js 16 [VERIFIED: nextjs.org/docs/app/api-reference/file-conventions/proxy]
- `auth.api.getSession()` requires Node.js runtime (it connects to PostgreSQL via Drizzle) — compatible
- Staging bypass MUST be checked FIRST (D-08) before the session check
- STAGING_KEY check is env-var-gated (not NODE_ENV-gated) per D-07
- Export function must be named `proxy` (not `middleware`)

```typescript
// Source: https://better-auth.com/docs/integrations/next + nextjs.org/docs proxy docs
// proxy.ts
import { NextRequest, NextResponse } from "next/server"
import { auth } from "@/auth"
import { headers } from "next/headers"

const PUBLIC_ROUTES = ["/login", "/register"]
const AUTH_ROUTES = ["/login", "/register"]

export async function proxy(request: NextRequest) {
  const path = request.nextUrl.pathname

  // D-07/D-08: Staging bypass FIRST, before any auth check
  if (
    process.env.STAGING_KEY &&
    request.headers.get("x-staging-key") === process.env.STAGING_KEY
  ) {
    return NextResponse.next()
  }

  // Session check — Node.js runtime, DB query via Drizzle adapter
  const session = await auth.api.getSession({
    headers: request.headers,
  })

  const isAuthenticated = !!session?.user
  const isPublicRoute = PUBLIC_ROUTES.includes(path)
  const isAuthRoute = AUTH_ROUTES.includes(path)

  // Redirect authenticated users away from auth pages
  if (isAuthRoute && isAuthenticated) {
    return NextResponse.redirect(new URL("/dashboard", request.nextUrl))
  }

  // Redirect unauthenticated users from protected pages
  if (!isPublicRoute && !isAuthenticated) {
    return NextResponse.redirect(new URL("/login", request.nextUrl))
  }

  return NextResponse.next()
}

export const config = {
  matcher: ["/((?!api|_next/static|_next/image|favicon.ico|.*\\.png$).*)"],
}
```

### Pattern 6: DAL `verifySession()` (Authoritative Auth Check)

**What:** Server-side session check called at the top of every DAL function.
**When to use:** ALL DAL functions call this before any DB query. Not optional.

```typescript
// Source: https://better-auth.com/docs/concepts/session-management + nextjs.org/docs/app/guides/authentication
// lib/dal/auth.ts
import "server-only"
import { cache } from "react"
import { headers } from "next/headers"
import { redirect } from "next/navigation"
import { auth } from "@/auth"

export const verifySession = cache(async () => {
  const session = await auth.api.getSession({
    headers: await headers(),
  })
  if (!session?.user) {
    redirect("/login")
  }
  return {
    userId: session.user.id,
    email: session.user.email,
    subscriptionPlan: (session.user as any).subscriptionPlan as "free" | "basic" | "pro",
    role: (session.user as any).role as "user" | "admin",
  }
})
```

**Note on session reactivity after Server Actions:** There is a known issue [VERIFIED: github.com/better-auth/better-auth/issues/3608] where server-side auth actions (sign-in, sign-out) don't automatically trigger `useSession()` to revalidate. After `signOutAction`, use `router.refresh()` or `authClient.useSession()` will pick up the cleared cookie on next render. This is acceptable for Phase 2 since the redirect to `/login` causes a full page navigation.

### Anti-Patterns to Avoid

- **Using `getServerSession()` anywhere:** This is NextAuth v5 API. Better Auth uses `auth.api.getSession()`. Zero imports of `getServerSession`.
- **Importing `auth` in Client Components:** `auth.ts` uses Node.js modules (`pg`, Drizzle). Only import it in server files.
- **Calling `auth.api.*` without `headers: await headers()`:** The session cookie is read from headers. Without them, session is always null.
- **Using Better Auth client SDK in Server Actions:** The `authClient` is for browser use only. Server Actions must use `auth.api.*`.
- **Missing `nextCookies()` as last plugin:** Without it, session cookies are not set in Server Actions, and the browser never receives the session cookie after login/registration.
- **Using `z.string().email()` (Zod v3 API):** In Zod v4 this must be `z.email()` (top-level function).
- **Passing `name` field from form to signUpEmail:** The `name` field in the form is suppressed (D-01). Pass `email` as the `name` value to satisfy Better Auth's required field requirement.
- **Running `drizzle-kit push`:** Never in any deployed environment. Always `generate` + `migrate`.
- **Checking `NODE_ENV !== 'production'` for staging bypass:** D-07 specifies env-var-gated bypass, not NODE_ENV-gated. Railway staging environments have `NODE_ENV=production`.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Password hashing | Custom bcrypt/argon2 implementation | Better Auth (scrypt by default) | OWASP-recommended algorithm, key stretching, salt management built-in |
| Session cookie management | Manual JWT signing, cookie serialization | Better Auth session system + `nextCookies()` | HttpOnly, Secure, SameSite, rotation, expiry all managed; edge cases with clock skew, replay etc. |
| CSRF protection | Custom CSRF token | Better Auth (built-in for mutations) | Correct SameSite cookie configuration covers 99% of cases |
| "Remember me" persistence | Custom cookie expiry logic | Better Auth default session persistence | Sessions already persist through browser restart by default |
| User enumeration protection | Custom conditional error messages | Return identical generic message in all error cases (D-05, D-06) | Simpler — same error message for all auth failures, no branching |

**Key insight:** Better Auth's server API handles all session lifecycle complexity. The only application code needed is: configure it, create the route handler, call `auth.api.signInEmail` / `auth.api.signUpEmail` from server actions, and call `auth.api.getSession` to read the session.

## Common Pitfalls

### Pitfall 1: `name` Field Required by Better Auth on Sign-up

**What goes wrong:** `auth.api.signUpEmail` requires a `name` field in the body. The project decision (D-01) collects only email + password. Passing `name: undefined` or omitting it causes a 422/validation error from Better Auth.
**Why it happens:** Better Auth's user schema has `name` as a required column. This is a library-level constraint not configurable without patching.
**How to avoid:** Pass `email` as the value for `name` in the `signUpAction`. The `name` column will be populated with the email address as a placeholder until Phase 7 adds `firstName`/`lastName`.
**Warning signs:** Sign-up returns an error even with valid email+password; Better Auth logs show "name is required".

### Pitfall 2: `nextCookies()` Plugin Position

**What goes wrong:** If `nextCookies()` is not the LAST plugin in the `plugins` array, cookie headers may not be forwarded correctly in Server Actions, causing the session to not be set after login/registration.
**Why it happens:** Better Auth processes plugins in order; `nextCookies` wraps the response to inject Set-Cookie headers — it must see the final response.
**How to avoid:** Always declare `plugins: [nextCookies()]` as the only (or last) entry.
**Warning signs:** Login succeeds (no error), redirect to dashboard fires, but the dashboard immediately redirects back to `/login` because no session cookie was set.

### Pitfall 3: Zod v4 `z.string().email()` Removed

**What goes wrong:** If using `z.string().email()` (Zod v3 pattern), Zod v4 will either throw a build error or silently fail depending on version. Installed version is 4.3.6.
**Why it happens:** In Zod v4, email validation moved from a string method to a top-level function: `z.email()`.
**How to avoid:** Use `z.email()` in all schemas. Also: `{ message: "..." }` becomes `{ error: "..." }` in Zod v4.
**Warning signs:** TypeScript error "Property 'email' does not exist on type ZodString"; or validation never fails for invalid emails.

### Pitfall 4: Proxy.ts Calls DB on Every Request (Performance)

**What goes wrong:** `auth.api.getSession()` in `proxy.ts` makes a DB call to validate the session token on EVERY request, including static assets and Next.js internal routes, if the matcher is too broad.
**Why it happens:** Better Auth stores sessions in the database (not purely JWT). The Drizzle adapter checks the `session` table on each call.
**How to avoid:** The existing `config.matcher` already excludes `_next/static`, `_next/image`, `favicon.ico`, and `.png` files. Verify matcher is correctly scoped. The DB call is lightweight (indexed session token lookup) but should not run on asset requests.
**Warning signs:** Slow page loads where every navigation incurs a visible DB roundtrip latency.

### Pitfall 5: Server Action Session Not Updating Client `useSession()`

**What goes wrong:** After `signOutAction()` (which is a Server Action), the client-side `authClient.useSession()` in `topbar.tsx` still shows the old session.
**Why it happens:** Server Actions do not automatically trigger client-side state revalidation. The `useSession()` hook in Better Auth does not listen for server-side session changes.
**How to avoid:** Rely on full page navigation (redirect) after auth actions. Since `signOutAction` redirects to `/login`, the browser navigation clears the topbar session state. For signIn, `autoSignIn: true` plus redirect to `/dashboard` achieves the same effect.
**Warning signs:** Topbar still shows logged-in user after logout; stale user data displayed after sign-in via a Server Action.

### Pitfall 6: Custom `additionalFields` Not in Session TypeScript Types

**What goes wrong:** `session.user.subscriptionPlan` and `session.user.role` are `undefined` at TypeScript level even though they exist at runtime, causing implicit `any` or type errors in DAL code.
**Why it happens:** Better Auth's TypeScript inference for `additionalFields` does flow through to session types, but the exact mechanism requires importing `auth` as a type reference on the client side, and on the server side the `auth.api.getSession()` return type may not automatically include the extra fields.
**How to avoid:** In `lib/dal/auth.ts`, cast the `session.user` to include the custom fields. For the Better Auth client, Better Auth's TypeScript inference propagates custom fields through when `typeof auth` is used as type parameter in `createAuthClient`. [MEDIUM confidence — verify TypeScript output after generation]
**Warning signs:** `subscriptionPlan` type is `never` or missing; TypeScript complains about accessing `.subscriptionPlan` on session user.

### Pitfall 7: Staging Bypass Leaks to Production

**What goes wrong:** If `STAGING_KEY` is accidentally set in the production Railway environment, the staging bypass becomes active and any request with the correct header bypasses auth entirely in production.
**Why it happens:** D-07 uses env-var presence (not NODE_ENV) to gate the bypass.
**How to avoid:** Do NOT set `STAGING_KEY` in the Railway production service. Set it only on staging/preview environments. Document this in `.env.example` comments (already present).
**Warning signs:** Production app allows unauthenticated access with the `x-staging-key` header.

```typescript
// Source: https://better-auth.com/docs/installation + https://better-auth.com/docs/concepts/database
// auth.ts
import { betterAuth } from "better-auth"

  emailAndPassword: {
    enabled: true,
    minPasswordLength: 8,
    autoSignIn: true,
  },
  database: drizzleAdapter(db, { provider: "pg" }),
  user: {
    additionalFields: {
      subscriptionPlan: {
        type: ["free", "basic", "pro"] as const,
        required: false,
        defaultValue: "free",
        input: false,
      },
      role: {
        type: ["user", "admin"] as const,
        required: false,
        defaultValue: "user",
        input: false,
      },
    },
  },
  plugins: [nextCookies()],  // Must be last
})
```

Better Auth generates four tables. [CITED: better-auth.com/docs/concepts/database]

- `user`: `id`, `name`, `email`, `emailVerified`, `image`, `createdAt`, `updatedAt`, + `subscriptionPlan`, `role` (via additionalFields)
- `session`: `id`, `userId`, `token`, `expiresAt`, `ipAddress`, `userAgent`, `createdAt`, `updatedAt`

The CLI command `npx @better-auth/cli@latest generate` generates the Drizzle-compatible TypeScript schema. For PostgreSQL provider, column types use `pgTable`, `varchar`, `boolean`, `timestamp`, `text`, and `pgEnum` from `drizzle-orm/pg-core`.
