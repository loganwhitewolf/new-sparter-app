## Auth Form Interaction Contract

### Layout — Both Login and Register Pages

- Container: `(auth)` layout shell — centered vertically + horizontally, `max-w-sm`, inherited from Phase 1
- Logo: Sparter wordmark above form — already rendered by `(auth)/layout.tsx`
- Form structure (top to bottom):
  1. Page heading (h1): "Accedi" or "Crea account" — 20px semibold, text-center
  2. Descriptor line: 14px regular, `text-muted-foreground`, text-center
  3. Error banner (conditional): `Alert variant="destructive"` — renders only when an auth error is present, positioned between descriptor and form fields
  4. Email input: `Input type="email"`, full width, `autoComplete="email"`
  5. Password input: `Input type="password"`, full width, `autoComplete="current-password"` (login) / `"new-password"` (register)
  6. Submit button: `Button variant="default"`, full width (`w-full`)
  7. Navigation link: text + anchor to the other auth page — 14px, text-center

No form wrapper card (`Card`) — the form sits directly on the white background, matching the Phase 1 stub pattern.

**Primary visual anchor:** the full-width emerald submit button at the base of the form.

**Source:** Phase 2 CONTEXT.md D-03, D-04; Phase 1 `(auth)/layout.tsx` inspection

### Register Page Specifics

- Route: `app/(auth)/register/page.tsx` (new file — stub did not exist in Phase 1)
- Page heading: "Crea account" — 20px semibold
- Descriptor line: "Inserisci email e password per creare il tuo account." — 14px muted
- Fields: email + password only (no name fields — D-01)
- Submit button: "Registrati"
- Navigation link below button: "Hai già un account? Accedi" — link portion navigates to `/login`
- Post-submit: redirect to `/dashboard` on success (D-02)

### Login Page Specifics

- Route: `app/(auth)/login/page.tsx` (existing stub — to be wired)
- Page heading: "Accedi" — 20px semibold
- Descriptor line: "Inserisci le tue credenziali per accedere." — 14px muted (already present in stub)
- Fields: email + password
- Submit button: "Accedi"
- Navigation link below button: "Non hai un account? Registrati" — link portion navigates to `/register`
- Post-submit: redirect to `/dashboard` on success

### Error Banner

- Component: `Alert` with `variant="destructive"` from shadcn/ui (install required)
- Position: between descriptor line and first input field
- Visibility: hidden when no error; renders (no animation) when error is present
- Width: full width of form container (`w-full`)
- Content: icon (AlertCircle from lucide-react, 16px) + error message text (14px)
- No dismiss/close button — error clears on next submit attempt

**Source:** Phase 2 CONTEXT.md D-04

### Loading State During Submit

- Pattern: disable the submit Button + show Loader2 spinner icon inside it
- Implementation: `Button disabled` with `<Loader2 className="mr-2 h-4 w-4 animate-spin" />` prefix
- Button text unchanged during loading: "Accedi" / "Registrati" (spinner appears before text)
- All inputs remain enabled during submit (do not lock the form)
- Duration: from form submit until server response received

**Source:** Claude's discretion — standard shadcn/ui loading pattern using lucide Loader2

### Input Validation

- Strategy: **submit-only** — no inline per-field validation, no real-time feedback
- Error display: single error banner (see above) for all error states
- No red borders on individual inputs — the banner is the sole error indicator
- Client-side Zod validation runs on submit before network call (fast fail for empty fields / password < 8 chars)

**Source:** Phase 2 CONTEXT.md D-04 (banner inline, no per-field errors), D-11 (password min 8 chars)

### Password Field

- Type: `Input type="password"` — browser-native masking
- No show/hide toggle — not required for v1 (adds complexity, not in scope)
- `autoComplete="current-password"` on login, `autoComplete="new-password"` on register

**Source:** Claude's discretion — omission is intentional, keeps form minimal

### Navigation Links

- Login → Register: "Non hai un account? Registrati" — plain text + `<Link>` (Next.js)
- Register → Login: "Hai già un account? Accedi" — plain text + `<Link>`
- Styling: 14px, text-center, `text-muted-foreground` for the static part, `text-foreground underline underline-offset-4` for the link part (hover: `text-primary`)

## Topbar Session Wiring Contract

### Avatar Fallback Strategy

When `firstName` and `lastName` are not yet set (standard in Phase 2 — profile is Phase 7):

1. Primary: `AvatarImage src=""` will fail — falls back to `AvatarFallback` automatically
2. Fallback text: **first character of the user's email address**, uppercased
   - Example: `loganwhitewolf@gmail.com` → fallback letter "L"
   - Implementation: `session.user.email.charAt(0).toUpperCase()`
3. Fallback styling: `bg-primary text-primary-foreground text-xs font-medium` (emerald background, white text) — matches Phase 1 stub

### Topbar Label Content

Replace Phase 1 placeholder content with real session data:

| Element | Phase 1 Placeholder | Phase 2 Real Value |
|---------|---------------------|---------------------|
| Name line (bold) | "Utente" | `session.user.email` — display full email as name until Phase 7 adds firstName/lastName |
| Email line (muted) | "utente@example.com" | `session.user.email` — same value (intentional duplication until Phase 7) |

**Rationale:** Phase 7 introduces `firstName`/`lastName`. Until then, email is the only identifier. Showing email in both slots is intentional and avoids an empty "Utente" placeholder in production.

**Source:** Phase 2 CONTEXT.md specifics note on topbar; Phase 1 `topbar.tsx` inspection

## Implementation Notes for Executor

1. **Alert component install:** Run `npx shadcn@latest add alert` before implementing error banners. This is the only new component required in Phase 2.

2. **Error banner placement:** Render `<Alert variant="destructive">` as a conditional block between the descriptor paragraph and the first `<Input>`. Use a state variable (`errorMessage: string | null`) — when null, render nothing (not `display:none`, just `{errorMessage && <Alert>}`).

3. **Loading spinner:** Import `Loader2` from `lucide-react`. Wrap it in `<Loader2 className="mr-2 h-4 w-4 animate-spin" />` as the first child of `<Button>` when `isPending` is true. Keep button text visible alongside spinner.

4. **Avatar fallback:** In `topbar.tsx`, replace hardcoded `"U"` with `session?.user?.email?.charAt(0).toUpperCase() ?? "U"`. The `?? "U"` guard handles edge cases where session is momentarily undefined.

5. **Topbar name display:** Show `session.user.email` in both the bold name slot and the muted email slot. This is correct behavior for Phase 2 — Phase 7 will update to firstName/lastName when those fields are populated.

6. **Form submit pattern:** Use React `useTransition` or `useActionState` (Next.js 16 App Router pattern) for server action submission. `isPending` drives the loading state on the Button.

7. **Link styling:** Use Next.js `<Link>` for `/login` ↔ `/register` navigation. Apply `className="underline underline-offset-4 hover:text-primary"` to the clickable word only (not the full sentence).

8. **No Card wrapper:** The auth form sits directly on white background — no `<Card>` wrapping. This matches the Phase 1 stub and `(auth)/layout.tsx` structure. Do not add a card border/shadow around the form.

9. **No per-field error styling:** Do not add `aria-invalid` or red border to individual inputs. The single error banner at the top of the form is the only error indicator.

10. **Password autoComplete:** Login password field: `autoComplete="current-password"`. Register password field: `autoComplete="new-password"`. This is required for browser password managers to work correctly.
