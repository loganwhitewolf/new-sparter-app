Create lib/dal/auth.ts with the verifySession() function. This is the authoritative session check — every DAL function in Phase 3+ calls this before touching the database.

```typescript
import 'server-only'
import { cache } from 'react'
import { headers } from 'next/headers'
import { redirect } from 'next/navigation'
import { auth } from '@/auth'

export const verifySession = cache(async () => {
  const session = await auth.api.getSession({
    headers: await headers(),
  })
  if (!session?.user) {
    redirect('/login')
  }
  return {
    userId: session.user.id,
    email: session.user.email,
    subscriptionPlan: (session.user as any).subscriptionPlan as 'free' | 'basic' | 'pro',
    role: (session.user as any).role as 'user' | 'admin',
  }
})
```

Implementation notes:
- `import 'server-only'` — MUST be first line (CLAUDE.md rule for lib/dal/ files)
- `cache()` from React deduplicates the `getSession` DB call within a single request lifecycle. Multiple DAL calls in one request hit DB once.
- `auth.api.getSession({ headers: await headers() })` — reading headers() here gives the current request's cookies containing the session token.
- `(session.user as any).subscriptionPlan` — TypeScript cast required because Better Auth's `additionalFields` may not auto-infer into the return type (Pitfall 6). The cast is safe because Better Auth guarantees the field is present when defined in `additionalFields`.
- Return type includes `userId` (D-10 — required for all Phase 3+ DAL queries), `email`, `subscriptionPlan`, and `role`.
- If session is null/undefined: `redirect('/login')` — never returns `null` to caller.

```typescript
'use server'
import { auth } from '@/auth'
import { headers } from 'next/headers'
import { redirect } from 'next/navigation'
import { LoginSchema, RegisterSchema } from '@/lib/validations/auth'

export type AuthActionState = { error: string | null }

export async function signInAction(
  _prev: AuthActionState,
  formData: FormData
): Promise<AuthActionState> {
  const parsed = LoginSchema.safeParse({
    email: formData.get('email'),
    password: formData.get('password'),
  })
  if (!parsed.success) {
    // D-05: generic message — Zod error treated same as auth error (no field-level leakage)
    return { error: 'Credenziali non valide. Riprova o contatta il supporto.' }
  }
  try {
    await auth.api.signInEmail({
      body: { email: parsed.data.email, password: parsed.data.password },
      headers: await headers(),
    })
  } catch {
    // D-05: generic message — never reveal whether email exists
    return { error: 'Credenziali non valide. Riprova o contatta il supporto.' }
  }
  redirect('/dashboard')
}

export async function signUpAction(
  _prev: AuthActionState,
  formData: FormData
): Promise<AuthActionState> {
  const parsed = RegisterSchema.safeParse({
    email: formData.get('email'),
    password: formData.get('password'),
  })
  if (!parsed.success) {
    // D-06: generic message — no field-level error details
    return { error: 'Si è verificato un errore. Riprova.' }
  }
  try {
    await auth.api.signUpEmail({
      body: {
        email: parsed.data.email,
        password: parsed.data.password,
        name: parsed.data.email, // Pitfall 1: Better Auth requires 'name' — use email as placeholder (D-01)
      },
      headers: await headers(),
    })
  } catch {
    // D-06: generic message — do not reveal email already registered
    return { error: 'Si è verificato un errore. Riprova.' }
  }
  redirect('/dashboard') // D-02: auto-login via autoSignIn:true → redirect directly to dashboard
}

export async function signOutAction(): Promise<void> {
  await auth.api.signOut({ headers: await headers() })
  redirect('/login')
}
```

Critical implementation rules:
- `'use server'` MUST be the first line of the file
- NEVER use Better Auth client SDK (`authClient.signIn`) in Server Actions — use `auth.api.*` only
- `name: parsed.data.email` is intentional (Pitfall 1) — Better Auth requires `name` field; D-01 says no name at signup
- NEVER call `cookies().set()` manually — `nextCookies()` plugin in auth.ts handles this automatically
- Error messages are EXACT Italian strings from UI-SPEC Copywriting Contract:
  - Login error: "Credenziali non valide. Riprova o contatta il supporto."
  - Register error: "Si è verificato un errore. Riprova."
- Zod parse errors return the SAME generic message as auth errors (D-05/D-06 — prevents enumeration)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser form → Server Action | FormData from browser — untrusted input, validated with Zod before any auth call |
| Server Action → Better Auth API | auth.api.* calls — internal Node.js, no external network hop |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-03-01 | Information Disclosure | signInAction error response | mitigate | D-05: identical generic error for Zod failure and auth failure — attacker cannot distinguish valid email from invalid (user enumeration prevention) |
| T-2-03-02 | Information Disclosure | signUpAction error response | mitigate | D-06: identical generic error for duplicate email and other failures — email existence not revealed |
| T-2-03-03 | Elevation of Privilege | scrypt password hashing | mitigate | Better Auth uses scrypt by default — never hand-rolled; slow hash prevents brute force acceleration |
| T-2-03-04 | Elevation of Privilege | verifySession() session check | mitigate | All Phase 3+ DAL functions call verifySession() first — unauthorized DB access blocked at DAL boundary, not just proxy.ts |
| T-2-03-05 | Tampering | CSRF on signInAction/signUpAction | mitigate | Server Actions use POST-only; Better Auth session cookie has SameSite=lax — cross-site POST blocked by browser |
| T-2-03-06 | Elevation of Privilege | session fixation | mitigate | Better Auth creates new session token on sign-in (token rotation built-in) — old session tokens invalidated |
</threat_model>
