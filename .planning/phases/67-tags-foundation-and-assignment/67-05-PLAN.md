<objective>
Build the TAG-03 backend: a single date-range matcher shared by both suggestion triggers (tag
creation and every subsequent import), with D-09's inclusive-boundary semantics, D-10's
already-tagged dedup, and the two thin server actions the create-time modal (Plan 67-08) and
post-import summary block (Plan 67-09) will call.

Purpose: D-08's two triggers are really one computation invoked from two places — re-scanning a
tag's date range against the user's transactions and filtering out already-tagged ones. Building
that computation once here, proven against the boundary/empty/ordering edges, means the two UI
plans need only wire a button, not re-derive matching logic.

Output: `lib/dal/tag-suggestions.ts`, `lib/services/tag-suggestions.ts`,
`lib/actions/tag-suggestions.ts`, covered by unit tests.
</objective>

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-67-12 | Information Disclosure | `getNewTagSuggestionsAction` | medium | mitigate | `getTag`'s userId-scoped lookup means a forged `tagId` for another user's tag returns `null` (mapped to an empty result), never leaking that user's transaction dates. |
