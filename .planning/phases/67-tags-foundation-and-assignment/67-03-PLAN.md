<objective>
Build the TAG-01 backend vertical: userId-scoped DAL queries over the `tag` table, a service layer
that owns the D-02 case/whitespace-insensitive uniqueness guard (both a fast inline pre-check and
a DB-constraint-backed fallback for the race case), and the three thin server actions the
`/settings/tags` UI (Plan 67-08) will call.

Purpose: every tag CRUD surface — settings page, bulk-assign dialog's tag list, detail-page
add/remove picker, suggestion flows — reads through `getTags`/`getTag` from this plan. Getting the
uniqueness guard right here (pre-check for fast UX + DB-catch for correctness under races) is what
makes D-02's "blocking inline error" promise actually hold under concurrent requests, not just in
the happy path.

Output: `lib/dal/tags.ts`, `lib/services/tag-operations.ts`, `lib/actions/tags.ts`, all covered by
unit tests using this codebase's established mocked-db pattern (no live DB in tests).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Client FormData → tag actions → service → DAL | Every mutating path re-derives `userId` from `verifySession()` server-side; no client-supplied `userId` is ever trusted. `tagId`/`id` values from the client are always re-scoped by `eq(tag.userId, userId)` in the DAL — a forged `tagId` belonging to another user resolves to `not_found`, never a cross-user mutation. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-67-06 | Tampering (IDOR) | `updateTagRow`/`archiveTagRow` | high | mitigate | Both DAL functions include `eq(tag.userId, userId)` in their WHERE clause — a `tagId` owned by another user resolves 0 rows, mapped to `TagMutationError('not_found')` by the service, never silently succeeding against someone else's row. |
| T-67-07 | Tampering | `CreateTagSchema`/`UpdateTagSchema` name field | low | mitigate | Zod `.min(2).max(60)` bounds; React renders tag names as text content everywhere (no `dangerouslySetInnerHTML` anywhere in this phase) — auto-escaped, no stored-XSS vector. |
| T-67-08 | Information Disclosure | Uniqueness pre-check timing | low | accept | The pre-check (`getTagByNormalizedName`) only ever runs inside an authenticated, already-`verifySession()`-gated action for the CALLER's own `userId` scope — it cannot be used to probe another user's tag names, since the query itself is userId-scoped. |
</threat_model>
