<objective>
Wire TAG-02's bulk-assign entry point into the transactions page: a new "Assegna tag" action in
the existing bulk-selection bar, a two-tab (Assegna/Rimuovi) multi-select dialog built from

select rows, assign or remove one or more tags in one action. It depends only on Plan 67-04's
already-ownership-verified assignment service, so it can ship independently of the CRUD settings
page (Plan 67-08) and the suggestion flows (Plan 67-05/67-09/67-08).

transactions table (row chips + dialog wiring), and an extended transactions page (data fetching).

## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Row-chip data flow | `getTagsForTransactionIds` (Plan 67-04) performs no ownership check of its own — it is only ever called in this plan with `transactionIds` already sourced from `getTransactions(filters)`, which IS userId-scoped internally; this plan must never call `getTagsForTransactionIds` with client-supplied ids directly. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-67-14 | Tampering | `BulkAssignTagsDialog`'s FormData construction | low | mitigate | `transactionIds`/`tagIds` are ultimately re-verified server-side by `bulkAssignTags`/`bulkRemoveTags` (Plan 67-04) regardless of what the client sends — a manipulated client payload cannot bypass ownership checks, only fail them. |
| T-67-15 | Tampering (XSS) | Tag name rendered in row chip / dialog checkbox label | low | mitigate | Tag names render as plain JSX text content (`{t.tagName}`, `{tag.name}`) — React escapes by default; no `dangerouslySetInnerHTML` anywhere in this plan's components. |
</threat_model>
