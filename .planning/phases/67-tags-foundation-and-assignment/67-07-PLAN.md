<objective>
Add a Tag section to the transaction detail page: current tags as removable chips, plus a picker
to add any not-yet-assigned tag — the D-07b single-transaction add/remove surface.

Purpose: the bulk-assign dialog (Plan 67-06) is the primary TAG-02 entry point, but a user viewing
one transaction's detail page needs a lightweight way to add or correct a single tag without
navigating back to the list and re-selecting. This plan is the second, narrower TAG-02 surface;
it shares no files with Plan 67-06 (different page, different client component) and can execute
in the same wave.

Output: an extended `TransactionDetailClient` with a Tag section, an extended detail RSC page
fetching the transaction's current tags and the full tag list, and an extended test suite.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|--------------|
| Detail-page single add/remove → transaction-tags actions | `addTransactionTagAction`/`removeTransactionTagAction` (Plan 67-04) independently re-verify that `transaction.id` and `tagId` both belong to the caller's `userId` — this client component's `transaction.id` comes from an already-ownership-verified RSC load (`getTransactionForDetail`), but the action layer does not skip its own check on that basis (defense in depth, same rationale as Phase 66's `T-66-12`). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|------------------|
| T-67-16 | Tampering (IDOR) | `handleAddTag`/`handleRemoveTag` | medium | mitigate | Both ultimately call Plan 67-04's `addTransactionTagAction`/`removeTransactionTagAction`, which re-verify ownership server-side — a manipulated client cannot tag/untag another user's transaction even by forging `transactionId`/`tagId` in the request. |
| T-67-17 | Tampering (XSS) | Tag chip / Select option text | low | mitigate | Tag names render as JSX text content throughout; no `dangerouslySetInnerHTML`. |
</threat_model>
