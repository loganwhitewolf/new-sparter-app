<objective>
Extend the existing `detachTransactionToDedicatedExpense` service (and its action + validation) so a single transaction can be isolated as a standalone expense with a chosen subcategory, on ANY transaction — including one whose expense holds only that single transaction. This is the backend slice of Phase 61: subcategory capture (STEXP-01), the single-transaction in-place re-hash branch that lifts the `SINGLE_TRANSACTION_EXPENSE` guard (STEXP-02), and the isolation property verified by test (STEXP-03).

Purpose: The netting-by-subcategory doctrine (option A, ADR 0004) is already usable, but a person's transfers all share one bank description and collapse into one `descriptionHash`-keyed expense — so a shared-subscription reimbursement cannot carry its own subcategory. This slice makes the existing detach capability able to (a) record the subcategory in one step and (b) work on a single-transaction expense without creating an orphan, per ADR 0016 decisions 2 and 4.

Output: A subcategory-aware, single-transaction-capable detach service with test coverage proving isolation from aggregation and Tier 2.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → server action (`detachTransaction`) | Untrusted `transactionId`, `title`, `subCategoryId` cross here; Zod-validated then session-scoped |
| server action → service (`detachTransactionToDedicatedExpense`) | Ownership must be enforced against the session `userId` before any write |
| service → DB (`expense`, `transaction`) | Writes must be scoped by `userId`; the in-place UPDATE is a new write path |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-61-01 | Elevation of Privilege / IDOR | in-place UPDATE on `expense` in `transaction-detach.ts` | high | mitigate | UPDATE is scoped by `expense.id AND expense.userId`; the loading SELECT already inner-joins `expense` on `expense.userId = input.userId`, so a foreign expense never loads. Covered by the existing TRANSACTION_NOT_FOUND test path. |
| T-61-02 | Tampering | client-supplied `subCategoryId` | medium | mitigate | Zod restricts to a positive integer in the action; the categorize action separately enforces `isSubCategoryVisibleToUser`. The detach service persists an FK-constrained integer; an invalid/foreign subcategory id fails the FK or is unreachable in the picker. If cross-user subcategory visibility becomes a concern, add a visibility check in a follow-up (noted, not required for the picker-driven flow). |
| T-61-03 | Repudiation | classification history on re-hash | low | accept | The in-place branch preserves the existing expense id, so `expenseClassificationHistory.expenseId` FKs remain valid; history is preserved, not orphaned. No new audit surface required. |
| T-61-SC | Tampering | npm/pip/cargo installs | high | mitigate | No new package installs in this phase (pure edits to existing files). No `[ASSUMED]`/`[SUS]` packages introduced; slopcheck N/A. |
