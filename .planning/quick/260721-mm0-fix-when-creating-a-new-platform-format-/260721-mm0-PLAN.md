<objective>
Fix a bug where creating a new private import format (platform format) with a secondary/additional description column silently drops that column — it is never saved to `importFormatVersion.secondaryDescriptionColumn`, so it is lost and cannot be used during import ("se creo un nuovo formato per una piattaforma con una descrizione aggiuntiva questa non viene aggiunta nella tabella e si perde nell'import").

Purpose: The combined `Primary — @secondary` description (used by `normalizeTransactionRow` in `lib/utils/import.ts`, e.g. for Satispay-style payer/payee context) never gets applied for private formats created through the wizard, even though the user explicitly configured it. The UI and service/DAL layers already fully support the field — the field is being dropped in the Server Action's form-parsing step, one layer up.

Output: `createPrivateImportFormatAction` correctly forwards `secondaryDescriptionColumn` from the submitted form to the Zod schema and downstream service; a regression test locks in the fix.
</objective>

<context>
@.planning/PROJECT.md
@CLAUDE.md

Root cause (confirmed by reading the full create-format flow):
- `components/import/import-format-wizard.tsx` (lines ~435-439) already renders a hidden input `name="secondaryDescriptionColumn"` wired to the `SelectField`, submitting `''` when "Nessuna" is chosen or the selected header otherwise. The UI is correct.
- `lib/validations/import.ts` `CreatePrivateImportFormatSchema` already declares `secondaryDescriptionColumn: OptionalImportFormatWizardColumnSchema` (empty string preprocesses to `undefined`). The schema is correct.
- `lib/services/import-format-wizard.ts` `createPrivateRows` already writes `secondaryDescriptionColumn: input.secondaryDescriptionColumn ?? null` into the `importFormatVersion` insert. The service/DAL write is correct.
- **The bug:** `lib/actions/import.ts` `createPrivateImportFormatAction` (lines 189-204) builds the object passed to `CreatePrivateImportFormatSchema.safeParse(...)` manually, field by field, from `FormData` — and never reads `formData.get('secondaryDescriptionColumn')`. Because the Zod field is `.optional()`, the missing key silently becomes `undefined` instead of failing validation, so the value the user picked in the wizard never reaches the schema, the service, or the database column. Every other optional wizard field (`amountColumn`, `positiveAmountColumn`, `negativeAmountColumn`) IS extracted via the existing `formString()` helper — `secondaryDescriptionColumn` was simply omitted from that list.

No other action builds this schema manually (`grep` confirms `CreatePrivateImportFormatSchema` is only referenced in `lib/services/import-format-wizard.ts` and `lib/actions/import.ts`), so this is the single point of failure — no additional edit-format/update-format action path exists for this schema.
