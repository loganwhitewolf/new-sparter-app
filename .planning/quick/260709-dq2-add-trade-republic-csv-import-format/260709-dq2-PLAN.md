# Quick Task 260709-dq2: Add Trade Republic CSV import format

## Goal

Add a **CSV** import format for Trade Republic (they now offer a CSV export, more
reliable than the PDF). The existing Trade Republic **PDF** contract stays as-is; the CSV
is a second format version for the same platform.

## Key facts (already verified against the codebase)

- Platform `trade-republic` already exists (`scripts/seed-data.ts` `platforms`), with a
  **PDF** format version 1 (`importFormatVersions`, synthetic headers
  `data,descrizione,importo_entrata,importo_uscita`).
- **No Drizzle migration is needed.** Every parsing-contract column already exists on
  `import_format_version`. This is purely a **seed-data addition** (a new format-version
  row), applied idempotently by `yarn db:seed` (`.onConflictDoNothing()` on the
  `(platformId, version)` unique key). The user asked for a "migration" but the correct
  mechanism here is a seed row, not a schema migration.
- `loadImportFormatsForDetection` (`lib/dal/import-formats.ts`) loads **all** active
  format versions as independent detection candidates (no one-per-platform collapse), so
  TR PDF v1 and TR CSV v2 coexist. For a CSV upload the PDF v1 required columns are
  absent → low confidence; the CSV v2 matches → selected. No collision.
- `parseBankDate` handles the ISO-8601 `datetime` column (micro/millisecond precision +
  `Z`) with `dateFormat: null`. `parseItalianAmount` handles the dot-decimal signed
  `amount` values (`-30.00`, `18.720000`) with `decimalReplace: false`.

## Real CSV header (reference — do NOT commit the user's file)

```
datetime,date,account_type,category,type,asset_class,name,symbol,shares,price,amount,fee,tax,currency,original_amount,original_currency,fx_rate,description,transaction_id,counterparty_name,counterparty_iban,payment_reference,mcc_code
```

Row types observed: CARD_TRANSACTION, BUY (savings-plan crypto), TRANSFER_INSTANT_INBOUND,
INTEREST_PAYMENT, DIVIDEND, TAX_OPTIMIZATION. All EUR. `amount` is the signed cash
movement for card/buy/transfer rows; savings-plan BUY descriptions carry
`... quantity: 0.000384`.

## Known limitations (document in SUMMARY, do NOT try to solve here)

- `amount` is the only column read. `fee`/`tax` are ignored, so INTEREST_PAYMENT and
  DIVIDEND rows import their **gross** amount (withholding tax not netted), and
  TAX_OPTIMIZATION rows (`amount = 0.000000`, real impact in `tax`) import as €0
  transactions. Acceptable for a first CSV contract; card/buy/transfer rows (the primary
  use) map exactly.
