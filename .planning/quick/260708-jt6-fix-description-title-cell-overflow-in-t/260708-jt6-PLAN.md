<objective>
Fix a CSS regression where long transaction/expense titles visually overflow
their table cell and overlap the columns to the right (Importo, Data,
Sorgente/Piattaforma). Reported via screenshot: since the na4 milestone task
widened `expense.title` to `text` and stopped truncating at write time
(commits c3b39af/19143a8), titles can now run well past 120 chars, and two
pre-existing CSS bugs — invisible while titles were short — now overflow
visibly.

Root cause (traced, not hypothesized):
1. **Transactions table**: `components/transactions/transaction-table.tsx`
   already uses `table-fixed` + a wrapper `overflow-hidden` (correct), and
   `TableCell className="max-w-0 w-full"` (the standard truncate-in-fixed-
   table trick already used elsewhere in this codebase). But the `<div
   className="flex flex-col gap-1">` that directly wraps `TransactionTitleEdit`
   inside that cell is missing `min-w-0`. Per CSS flexbox rules, a flex item
   defaults to `min-width: auto` (content-based) unless `min-w-0` resets it —
   without that reset at this level, the div can grow to its content's natural
   width regardless of `min-w-0` further down the tree, breaking the
   truncation chain and letting text spill into the next `<td>`.
2. **Expenses table**: `components/expenses/expense-table.tsx`'s `<Table>` has
   NO `table-fixed` class, and its wrapper `<div>` has no `overflow-hidden`.
   Every other `TableHead` in this table already has an explicit width
   (`w-32`, `w-24`, `w-36`, `w-44`, `w-10`) and the title column's `TableCell`
   already uses the same `max-w-0 w-full` trick as the (working) transactions
   table — but without `table-fixed` on the `<table>` element, that trick has
   no effect: `table-layout: auto` sizes columns from content, not from the
   `max-w-0`/`w-full` hints, so a long title's `ExpenseTitleEdit` span (which
   already has a correct min-w-0 chain) is never actually constrained.

Purpose: users must be able to scan the table without a long description/title
bleeding into adjacent columns. The visible fix must remain exactly what
already exists elsewhere in this codebase (ellipsis truncation via the
`truncate` Tailwind utility + the full text already wired to the native
`title=""` attribute for hover) — no new truncation mechanism is needed once
the two structural CSS bugs are fixed.
</objective>
