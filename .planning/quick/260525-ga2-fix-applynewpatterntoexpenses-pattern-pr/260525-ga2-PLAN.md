<objective>
Make pattern promotion from suggestions actually visible and verifiable:
1. Stop silently swallowing errors in the call sites of `applyNewPatternToExpenses` so any real bug surfaces in logs.
2. Add structured logging inside `applyNewPatternToExpenses` so we can confirm scan/match counts on each call.
3. Extend `revalidateCategorizationSurfaces` to also invalidate the `/import` route subtree so the suggestions page reflects the new state after promotion.

Purpose: After the prior fixes (cause-code unique-violation detection, soft-delete reactivation, removed LIMIT/added ORDER BY), the user still reports promoted patterns "not applied". Without logs we can't tell whether (a) the function is throwing and being swallowed, (b) zero expenses actually match the prefix regex, or (c) updates happen but the suggestions page still shows the pattern because its data isn't revalidated. This plan delivers the observability and revalidation needed to either confirm the fix works or pinpoint the remaining defect.

Output: Three modified server files plus a manual verification checkpoint where the user promotes a real suggestion and confirms (via server logs + suggestions page reload) that all uncategorized expenses matching the pattern are updated.
</objective>

In `lib/actions/revalidation.ts::revalidateCategorizationSurfaces`:
- Add `revalidatePath('/import', 'layout')` to the list of `revalidatePath` calls. The `'layout'` second argument tells Next.js 16 to invalidate the entire `/import` segment subtree, which includes `/import/[fileId]/suggestions` (where the promoted pattern's match group originated). Without this, the user navigates back to the suggestions page and still sees stale `uncategorizedTxs` because the inner `expense.subCategoryId` reads are cached.
- Keep the existing `revalidatePath(APP_ROUTES.expenses)`, `transactions`, `dashboard`, `categorySettings`, and `refresh()` calls unchanged.
- Add a one-line comment above the new call: `// Suggestions page reads expense.subCategoryId; promoted patterns clear suggestions when applied.`

Constraints:
- Do NOT add `/import` to `APP_ROUTES` unless it's already there — it is, but the `'layout'` revalidation flag is not encoded in `APP_ROUTES.import` (which is just the string). Pass `APP_ROUTES.import` as the first arg and `'layout'` as the second to keep route constants centralized.
- Do NOT change any callers of `revalidateCategorizationSurfaces`.
- Verify `revalidatePath`'s second argument accepts `'layout'` in this Next.js version before committing: check `node_modules/next/cache.d.ts` (`revalidatePath(originalPath: string, type?: 'layout' | 'page'): void` — it should).
