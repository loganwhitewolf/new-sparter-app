<objective>
Permanently fix two recurring bugs in onboarding step 4 ("categorizzazione") and document the expected behaviour so neither is re-fixed a third time.

Bug 1 — light theme regresses. `page.tsx` resolves the theme with an undefended magic-number ternary (`step === 4 ? "light" : "dark"`). The value is correct today but nothing guards it: any refactor of step wiring can silently drop it. Fix: extract a tiny pure exported function `onboardingThemeForStep(step)`, unit-test it, and document WHY step 4 is light.

Bug 2 — a catalogued item disappears. After categorizing, the action calls `revalidatePath('/onboarding')`; Step4 re-renders and `getTopUncategorizedExpenses` filters `sub_category_id IS NULL`, so the just-categorized expense is excluded, its `SubcategoryCombobox` unmounts, and the client-only green-check state is lost. Fix (robust, not minimal): Step4 loads the top-15 expenses regardless of categorized status (stable list) and renders already-categorized rows with the green check on the first server render. This survives revalidation AND manual refresh.

Purpose: stop the back-and-forth. The new test is the executable spec; code comments at the three touch points state the invariant.
Output: `onboardingThemeForStep` + theme test; `getTopExpensesForOnboarding` DAL query + shape test; Step4 + `SubcategoryCombobox` reworked to seed initial categorized state; comments + CONTEXT.md note.
</objective>

# Interface contracts (consumed by tasks below)

## lib/dal/transactions.ts — existing model to copy
`getTopUncategorizedExpenses(userId, limit=15)` runs raw `db.execute(sql\`...\`)`:
DISTINCT ON (description_hash), `total_amount::numeric < 0`, `sub_category_id IS NULL`,
ORDER BY description_hash, ABS(total_amount::numeric) DESC, LIMIT (hard-capped at 100),
then JS-side sort by |totalAmount| DESC. Row type `TopUncategorizedExpenseRow = { id, title, descriptionHash, totalAmount }`. Wrapped in react `cache()`.

## DB schema (drizzle)
- `expense` table `"expense"`: id (text), title (varchar), descriptionHash (`description_hash`), subCategoryId (`sub_category_id`, nullable int FK -> sub_category.id), totalAmount (`total_amount`, numeric string), userId (`user_id`).
- `subCategory` table `"sub_category"`: id (serial), name (varchar). Subcategory name for an expense = `LEFT JOIN sub_category sc ON sc.id = e.sub_category_id` -> `sc.name`.
  NOTE (non-goal): user subcategory overrides (`user_subcategory_override.custom_name`) are intentionally NOT joined here — the onboarding green-check label uses the canonical system name only. Keep it cheap.

## subcategory-combobox.tsx — current client contract
Props: `{ expenseId, expenseTitle, expenseAmount, categories, mostUsed }`. Internal `isCategorized` state starts `false`; when the action returns no error it flips to `true` and renders the green-check branch (rounded `bg-success/10` row with `CheckCircle` + title + amount). This branch is the durable target — we add props to seed it on first server render.

## lib/validations/onboarding.ts — current exports
`OnboardingStepSchema`, `STEP_NAMES`, `parseOnboardingStep(value)`. Theme mapping function will be added here (co-located with the step domain logic, server-only-free so it is unit-testable without rendering the page).
